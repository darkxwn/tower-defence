## 2. Комментарии

### 2.1 Стиль
- Четкие и понятные
- Без спецсимволов и символов Unicode

### 2.2 Формат размещения

**Для переменных, полей классов, констант** — справа, нижний регистр:
```cpp
int n = 0; // количество элементов
float speed = 1.5f; // скорость движения
```

**Для структур, перечислений, методов классов и функций** — на строке перед объявлением:
```cpp
// Расчитывает расстояние между двумя точками
float calculateDistance(sf::Vector2f a, sf::Vector2f b);
```

**Для блоков кода со сложной логикой** — перед блоком, кратко:
```cpp
// фильтрация уничтоженных снарядов
projectiles.erase(
    std::remove_if(projectiles.begin(), projectiles.end(),
        [](const Projectile& p) { return !p.isAlive(); }),
    projectiles.end()
);
```

**Для логических блоков внутри методов (отрисовка, расчёты, инициализация)** — перед блоком, нижний регистр:
```cpp
// полоска жизней — позиция и фон
sf::Vector2f barPos = mapOffset + pos + offset;
sf::RectangleShape bgBar({ barWidth, barHeight });
bgBar.setFillColor(Colors::Theme::HpBarBg);

// заполнение полоски пропорционально hp
float ratio = (float)health / (float)maxHealth;
sf::RectangleShape hpBar({ barWidth * ratio, barHeight });
hpBar.setFillColor(Colors::Theme::HpBarFill);

// спрайт врага
sf::Sprite sprite(ResourceManager::get(texName));
```

### 2.3 Запрещено
- Местоимения (я, мы, он, она, оно, они, мой, наш и т.д.)
- Слова: "важно", "можно", "нужно", "обязательно", "следует", "рекомендуется" и подобные
- Использование цифр (1, 2, 3, 4 и т.д.) для нумерации или внутри текста комментариев

### 2.4 Разрешено
- Оставлять todo-заметки на оптимизацию и последующую реализацию задач

## 3. Именование

| Элемент | Стиль | Пример |
|---------|-------|--------|
| Переменные | `camelCase` | `enemyCount`, `isAlive` |
| Поля классов | `camelCase` | `fireTimer`, `currentAngle` |
| Поля структур | `camelCase` | `gridPos`, `mapOffset` |
| Методы/функции | `camelCase` | `update`, `render`, `calculatePath` |
| Геттеры | `getCamelCase` | `getGridPos`, `getEndReason` |
| Сеттеры | `setCamelCase` | `setPosition`, `setSelectedSlot` |
| Константы | `CamelCase` | `MaxEnemies`, `DefaultSpeed` |

### 3.1 Комментарии к геттерам и сеттерам
- Геттеры: комментарий начинается с `Получение ...`
- Сеттеры: комментарий начинается с `Изменение ...`

Примеры:
```cpp
// Получение позиции на сетке
sf::Vector2i getGridPos() const;

// Изменение масштаба иконки
void setIconScale(sf::Vector2f scale);
```

## 4. Процесс планирования

### 4.1 Обязательные пункты PLAN.md
Каждая задача описывается со следующей структурой:

- **Текущая задача** — название и описание
- **Способы решения**:
  - Краткое описание каждого варианта
  - Плюсы и минусы
  - Выбранный вариант и обоснование
  - Примеры кода реализации
- **Этапы разработки** — пошаговое описание

### 4.2 Необязательные пункты
Записываются только при уместности:
- Промежуточные этапы
- Детали архитектуры для проверки
- Зависимости от других систем

### 4.3 Примечание
Любые непонятные задания/вопросы/задачи уточняются у пользователя перед выполнением.

## 5. Процесс разработки

| Шаг | Действие | Описание |
|-----|----------|----------|
| 1 | Описание задачи | Краткое описание что требуется сделать |
| 2 | Уточняющие вопросы | Вопросы к пользователю, переход к шагу 3 только после ответов |
| 3 | Создание `.hpp` | Заголовочные файлы с соблюдением комментариев и именования |
| 4 | Корректировки | Внесение правок пользователем при необходимости |
| 5 | Создание `.cpp` | Реализация методов к утверждённым `.hpp` файлам |
| 6 | Проверка соответствия | Сверка `.cpp` и `.hpp` на полное соответствие |
| 7 | Описание внедрения | Описание изменений в других файлах при интеграции |
| 8 | Внедрение | Интеграция написанного кода в проект с разрешения пользователя |
| 9 | Проверка корректности | Верификация кода из шага 8 |

## 6. Рабочий процесс

### 6.1 Git
- Простая модель — основная ветка, прямые коммиты

### 6.2 Существующий код
- Не приводить к требованиям при работе
- Новый код следует требованиям из этого документа

### 6.3 Todo-листы
- Использовать для отслеживания прогресса внутри задачи
- Обновлять статус в реальном времени

## 7. Разделители в заголовочных файлах

Для обозначения начала класса в `.hpp` файлах используется:
```cpp
///////////////////////////////////////////////////////////////////////////
//
// КЛАСС TOWER
//
///////////////////////////////////////////////////////////////////////////
```

## 8. История изменений — HISTORY.md

Файл `HISTORY.md` ведётся для сохранения контекста между сессиями.

Формат записи завершённой задачи:
```markdown
### [дата] — название задачи
- что сделано
- какие файлы затронуты
- примечания
```

Обновляется после завершения каждой задачи.
