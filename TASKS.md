# Задачи разработки

## 1. Создание нормального логгера и вывод ошибок через него

### Суть
Требуется централизованная система логирования для вывода информационных сообщений, предупреждений и ошибок. Текущий код содержит LOGE/LOGI/LOGW в Logger.hpp, но не везде используется. Нужно унифицировать вывод ошибок через логгер.

### Варианты решения

**Вариант A: Расширение текущего Logger**
- Добавить LEVELы: DEBUG, INFO, WARNING, ERROR
- Использовать макросы типа `LOGD()`, `LOGI()`, `LOGW()`, `LOGE()`
- Вывод в консоль + опционально в файл

**Вариант B: Использование spdlog или аналога**
- Подключить внешнюю библиотеку
- Больше возможностей (ротация файлов, форматы)

**Вариант C: Свой минимальный логгер**
- Простой класс с enum Level
- Потокобезопасный вывод

### Выбранный вариант и обоснование
**Вариант A** — расширение текущего Logger.hpp. Минимальные зависимости, полный контроль, соответствует стилю проекта.

## 3. Бесконечные волны

### Суть
Волны генерируются бесконечно, система сама определяет количество и порядок врагов на основе данных из .map файла (только типы врагов). При убийстве врагов игрок получает случайное количество денег в промежутке (будет зависеть от глобальных улучшений после их реализации). Money — **общее для всех уровней** (глобальное), не отдельное для каждого уровня.

### Варианты решения

**Вариант A: WaveSystem генерирует волны**
- В .map указываются только типы врагов: `enemies=basic,fast,armored`
- WaveSystem читает типы и генерирует волны по алгоритму
- Каждая волна увеличивает сложность (больше врагов, чаще спавн)

**Вариант B: КонфигWaveData**
- волны описываются в отдельном конфиге
- Но тогда не бесконечные

**Вариант C: Формула сложности**
- Волны растут по математической формуле
- Количество = base + wave * multiplier

### Выбранный вариант и обоснование
**Вариант A** — WaveSystem генерирует волны based on типах из .map. Гибкость, масштабируемость, соответствует требованию.

## 4. Система звёзд для уровней

### Суть
Для каждого уровня нужно 3 звезды. Требуется 2 спрайта: серая (неполученная) и желтая (полученная).

### Варианты решения

**Вариант A: Три отдельных спрайта**
- star-empty.png (серая)
- star-filled.png (желтая)
- Одна звезда = 1 спрайт

**Вариант B: Один спрайт с анимацией/маской**
- Все три звезды на одном изображении
- Рендер с учетом количества

**Вариант C: Контейнер UI::StarRating**
- Новый виджет-контейнер
- Внутри 3 Image элемента
- Меняет цвет программно

### Выбранный вариант и обоснование
**Вариант C** — виджет-контейнер. Гибкость, легко менять количество звезд, программная смена цвета (не нужны доп. спрайты).

### Как достигаются звёзды (нужно продумать)
- ★★★ — пройти уровент без потери жизней / с определенным количеством жизней
- ★★ — пройти уровень / не потерять башни
- ★ — просто пройти уровень

Конкретные условия: нужно определить после реализации механик.

## 5. Улучшение башен

### Суть
Два вида улучшений:
1. **В игре** — улучшение конкретной башни на уровне
2. **Глобальное** — улучшения в подменю, применяются ко всем башням определенного типа

### Варианты решения

**Вариант A: Сначала внутриигровые улучшения**
- Реализовать upgrade в меню башни (уже есть подменю Sell/Upgrade)
- Затем глобальные улучшения

**Вариант B: Сначала глобальные улучшения**
- Создать подменю Upgrades
- Определить глобальные уровни улучшений для каждого типа башни
- Затем внутриигровые

**Вариант C: Параллельно**
- Обе системы независимы
- Реализовать обе сразу

### Выбранный вариант и обоснование
**Вариант A** — сначала внутриигровые. Уже есть подменю (Sell/Upgrade), проще начать, видно результат. Глобальные улучшения строятся поверх.

### 5.1. Внутриигровые улучшения (в игре)

**Суть**: Игрок видит текущий уровень башни, может улучшить её за деньги.

**Реализация**:
- Добавить поле `level` в Tower
- В подменю добавить кнопку "Улучшить" с ценой
- При upgrade: увеличить статы (damage, range, fireRate)
- Визуально показывать уровень (звездочки или ранг)

### 5.2. Глобальные улучшения (подменю Upgrades)

**Суть**:Persistent улучшения, которые применяются ко всем башням определенного типа.

**Реализация**:
- В .map или отдельном конфиге: базовые статы для каждого типа башни
- Глобальный множитель для каждого типа
- Подменю Upgrades: показывает типы башен, уровни улучшений, цены
- Сохранение прогресса улучшений

## 6. Система сохранений

### Суть
Сохранение прогресса: звёзды по уровням, глобальные улучшения башен.

### Варианты решения

**Вариант A: JSON файл + SaveManager**
- saves/progress.json
- Отдельный класс SaveManager
- Человекочитаемый, легко редактировать

**Вариант B: Бинарный файл**
- Компактнее, быстрее
- Сложнее редактировать

**Вариант C: В SettingsManager**
- Расширить существующий SettingsManager
- Единая система настроек и сохранений

### Выбранный вариант и обоснование
**Вариант A** — JSON + SaveManager. Человекочитаемый, отдельно от настроек, не нужно менять SettingsManager.
